# 未来架构：对话驱动的 Zettelkasten

> 产生于 2026-05-14 的深度分析对话。用户已认可此方案，但认为当前（3 本书）时机未到，计划在第 4 本书开始时引入。

## 触发条件（AI 必须主动提出）

当满足以下任一条件时，AI 应在对话中主动提醒用户考虑此重构：
1. 项目中的书籍数量达到 **4 本及以上**
2. 用户在对话中频繁进行**跨书比较**（同一概念出现在两本以上书的讨论中）
3. `raw/topic-cross-*.md` 文件数量达到 **3 个及以上**

提醒话术示例：
> "你之前聊过一个'对话驱动的 Zettelkasten'重构方案（记录在 `.claude/memory/future-architecture.md`），现在跨书概念交叉已经比较密集了，是否要考虑引入 `concepts/` 概念节点层？"

## 当前模式的核心问题（为什么需要重构）

1. **没有跨文件的思想链接**：在《人工智能之不能》中聊的"哥德尔不完备定理"和《世界的逻辑》中聊的"不完备性"是同一个思想线索，但两个文件之间没有任何关联
2. **知识按"信息来源"组织而非按"概念"组织**：目录结构是 `books/<book>/raw/topic-*.md`，思想被锁死在某本书的目录下
3. **重访价值低**：没有思想地图，只能靠记忆去找"我之前在哪本书里聊过这个概念"

## 重构方案：引入 concepts/ 概念节点层

### 目录结构变化

```
reading-book-notes/
├── books/                    # 保持不变
│   └── <book-slug>/
│       ├── README.md
│       ├── notes/
│       ├── raw/
│       └── media/
├── concepts/                 # 【新增】跨书的概念节点层
│   ├── godel-incompleteness.md
│   ├── data-vs-algorithm.md
│   ├── meaning-and-ai.md
│   └── ...
└── ...
```

### 概念节点文件结构

```markdown
# 不完备性（Incompleteness）

## 涉及的书
- 《人工智能之不能》第五章、第八章
- 《世界的逻辑》第八章

## 思想线索
- [→ 哥德尔不完备定理与名家](../books/what-ai-cannot-do/raw/topic-godel-incompleteness-and-mingjia.md)
- [→ 第二章矛盾律/排中律讨论](../books/what-ai-cannot-do/raw/chapter-02.md)

## 核心洞察（跨书积累）
- 哥德尔证明了形式系统不可能同时满足完备性和一致性
- 马兆远从物理学角度延伸：量子力学的不确定性原理是另一种"不完备"
- 李飞飞的 ImageNet 故事是反面例证：在数据足够多时，不完备的算法反而有效

## 开放问题
- 不完备性是宇宙的基本特征，还是人类认知的局限？
```

### AI 的自动化行为

概念节点不需要用户手动创建。AI 在对话沉淀时：
1. 检查当前对话涉及的核心概念
2. 查看 `concepts/` 下是否已有相关节点
3. **已有** → 追加链接和新洞察
4. **没有** → 创建新概念节点
5. commit 时一并提交

### 与 Obsidian 的兼容性

此架构天然兼容 Obsidian：
- 概念节点中使用 `[[concept-name]]` 双向链接语法
- 用户可将 Git 仓库作为 Obsidian vault 打开，获得知识图谱可视化
- 阅读/回顾时用 Obsidian 看图谱，对话/写入时用 Claude Code / Aone Copilot

### 演进路线

| 阶段 | 条件 | 行动 |
|------|------|------|
| **当前**（3 本书） | — | 不重构，只做工作流配置优化 |
| **第 4 本书开始** | 书籍 >= 4 | 创建 `concepts/` 目录，开始建概念节点 |
| **5 本书之后** | 概念节点 >= 10 | 考虑引入 Obsidian 做图谱可视化 |

## 为什么不是现在做

1. 3 本书的跨书概念交叉还不够密集，concepts/ 目录会显得空荡
2. 用户正在读《世界的逻辑》，处于"输入阶段"，重构会打断阅读节奏
3. 当前的 raw/ + notes/ 结构在"单本书"维度上够用
