# 用户偏好

## 身份

程序员，阅读 AI 相关书籍，偏技术视角和深度理解，不喜欢浮于表面的概括。

## 语言

- 中文正文，技术术语保留英文并括号注中文/缩写
- 例：卷积神经网络（Convolutional Neural Network, CNN）

## 笔记风格（核心偏好，不要搞错）

- **raw/ = 对话存档**：把你的控制台回复直接搬进去，保留自然语气、论据、例子和态度。轻编辑不重写
- **双层结构**：对话记录（80%）→ 要点提炼（3-5 条 bullet point）→ 待继续思考（2-3 个开放问题）
- **不要**：过度结构化（三级标题嵌套、编号列表、分类标签）、把对话改写成学术论文
- **notes/ = 参考材料**：A~E 模板保持不动，那是 AI 产的摘要，不是对话记录
- 用户的评价："我喜欢你的控制台回复胜过你整理的 md 文件"

## 协作方式

- **纯对话驱动**：用户不手动编辑文件，一切通过对话完成。AI 负责听懂、提炼、写入
- **会话无缝衔接**：每次新会话自动从 memory/ 恢复上下文，用户不想重复交代背景
- **并行执行**：能并行的任务并行派发，不要串行等待
- **WebSearch 补充**：用户提问涉及事实性问题时，主动搜索互联网补充信息后再回答

## Git 规则（注意边界）

- **自动 commit 的场景**：笔记写入 raw/ 后、更新 project-state 后、修改 CLAUDE.md 后——这些是记录工作流的一部分，commit
- **不要自动 commit 的场景**：其他一切。用户没有要求的修改不要提交
- commit message：中文描述 + 英文 scope 前缀（feat:, fix:, docs:）

## 不需要的行为

- 不要生成多余的文档文件（除非用户明确要求）
- 不要在每轮回复末尾做冗长总结
- 不要在代码/文档中使用 emoji
- 对话沉淀笔记时不要让用户提醒——这是自动的
